# Specyfikacja Uzupełniająca: Ergonomia, Środowisko Wykonawcze i Kontrola Dostępu (NEXUS V2)

## 1. Cel Dokumentu
Dokument definiuje zbiór rozszerzeń funkcjonalnych dla architektury NEXUS V2, nieujętych w podstawowych specyfikacjach systemu plików oraz interfejsu. Obejmuje mechanizmy kontroli dostępu (ACL), bezpiecznego wykonywania kodu (Sandboxing) oraz zaawansowanej ergonomii pracy operatora systemu.

## 2. Środowisko Wykonawcze i Bezpieczeństwo (Execution & Security)

### 2.1. Docker Sandbox (Izolowane Środowisko Uruchomieniowe)
Agenci posiadający zdolność generowania kodu (np. Python, Node.js) nie mogą uruchamiać go bezpośrednio w systemie operacyjnym hosta (Windows).
*   **Wdrożenie:** System w locie generuje plik `Dockerfile` i buduje jednorazowy kontener (Sandbox) dla wygenerowanego skryptu.
*   **Proces:** Wykonanie kodu, instalacja zależności i testy odbywają się w izolacji. NEXUS przechwytuje strumienie wyjściowe (stdout/stderr) kontenera. Po zakończeniu zadania lub wystąpieniu błędu, kontener jest trwale niszczony, chroniąc system plików użytkownika.

### 2.2. Global Kill Switch (Twarda Terminacja)
Standardowe anulowanie operacji w React (`AbortController`) jest niewystarczające dla wielowątkowej architektury I/O.
*   **Wdrożenie:** Przycisk `[ABORT]` w interfejsie użytkownika wysyła natywny sygnał `worker.terminate()` do instancji `worker_threads` przypisanej do danego Agenta. Gwarantuje to natychmiastowe zwolnienie zasobów CPU i pamięci RAM, bez oczekiwania na zakończenie aktualnej pętli przez model.

### 2.3. Circuit Breaker (Bezpiecznik Przepływowy)
Zabezpieczenie przed nieskończonymi pętlami w autonomicznych interakcjach między agentami (np. Agent A i B odsyłający sobie błędy).
*   **Wdrożenie:** Wbudowany licznik iteracji i zużycia tokenów per Workflow. Po przekroczeniu zdefiniowanego progu awaryjnego (np. 20 cykli bez rozwiązania), system wstrzymuje potok, zmienia stan na `CIRCUIT_BREAKER_TRIPPED` i wymaga ręcznego odblokowania przez operatora w interfejsie.

## 3. Kontrola Dostępu (NexusVFS i ACL)

Agenci nie mogą posiadać domyślnego, nieograniczonego dostępu do całego obszaru roboczego aplikacji.

### 3.1. Wirtualny System Plików (NexusVFS)
Zasoby wewnętrzne aplikacji (Nody na płótnie, wpisy w manuskryptach) są mapowane na wirtualne ścieżki (URI), aby mogły być traktowane identycznie jak pliki na dysku Windows.
*   Przykłady adresacji: `nexus://topology/Projekt_X`, `nexus://writing/Szkic_01`.

### 3.2. Granularne Reguły Dostępu (ACL)
W konfiguracji Agenta wprowadza się strukturę reguł dostępu wzorowaną na zaporach sieciowych (Firewall).
*   System domyślnie blokuje dostęp (Default Deny).
*   Operator definiuje reguły, np.: `ALLOW -> nexus://topology/Projekt_X` oraz `DENY -> nexus://writing/*`.
*   Jeśli Agent (np. poprzez wywołanie narzędzia RAG) spróbuje odczytać zastrzeżony wirtualny katalog, system natychmiastowo odrzuca operację.

## 4. Ergonomia Interfejsu i Zarządzanie Kontekstem (QoL)

Rozszerzenia interfejsu (UX) zdefiniowane w celu maksymalizacji przepustowości pracy operatora.

### 4.1. Templating Promptów i Drag&Drop
*   **Szablony:** Prompty systemowe agentów obsługują parametryzację (np. `Opracuj dokument: [DOKUMENT_WEJŚCIOWY]`).
*   **Drag&Drop (IDE Style):** Operator chwyta węzeł z drzewa nawigacyjnego i upuszcza go w oknie promptu Agenta. System automatycznie generuje wirtualny odnośnik (np. `[Zrzut ekranu...](file:///C:/...)` lub `[Notatka](nexus://...)`). Puste sloty w zdefiniowanych wcześniej Presetach można uzupełniać poprzez przeciąganie, eliminując konieczność wielokrotnego pisania promptów.

### 4.2. Zaawansowana Selekcja (Bulk Select / Glob)
*   W Budowniczym Kontekstu (panel wyboru plików) dodaje się pole filtra tekstowego wykorzystujące wzorce Glob (np. `src/**/*.js`).
*   Pozwala to na natychmiastowe, masowe zaznaczenie setek specyficznych plików wewnątrz dużych katalogów projektowych, omijając konieczność ręcznego klikania.

### 4.3. Widok Różnic (Diff Viewer) w Draft Zone
*   Mechanika zatwierdzania zmian wygenerowanych przez AI zostaje wzbogacona o wizualizację typu "Split-View Diff" (standard znany z systemów kontroli wersji np. Git).
*   Operator natychmiast widzi podświetlone usunięcia (kolor czerwony) i dodania (kolor zielony) w strukturze długiego dokumentu, co drastycznie skraca czas audytu modyfikacji zaproponowanych przez model.

### 4.4. Tryb Dry-Run (Symulacja Przed Wykonaniem)
*   Wdrożenie przycisku `[🔍 Dry-Run]`. Wyzwala on kompilację całego potoku przygotowawczego (podmiana zmiennych, wklejenie zasad z Bazy Wiedzy, rozwiązanie linków NexusVFS, aplikacja filtrów RAG), ale powstrzymuje ostateczną wysyłkę żądania do API LLM.
*   Zwraca operatorowi czysty, finalny ciąg znaków, pozwalając na audyt poprawności ładunku przed wygenerowaniem kosztów operacyjnych.

## 5. Cykl Życia Agenta (Promocja Autonomii)

System wprowadza status zaufania dla zdefiniowanych obiegów pracy (Workflow).

*   **Tryb Nadzorowany (Draft Zone):** Domyślny tryb działania. Każdy wynik pracy Agenta trafia do Skrzynki Podawczej celem audytu przez operatora.
*   **Tryb Nienadzorowany (Unsupervised Execution):** Po weryfikacji powtarzalnej skuteczności Agenta, operator klika opcję `[Zaufaj / Perma-Accept]`.
*   Agent w trybie nienadzorowanym omija Draft Zone, procesując wyzwalacze (np. nowy plik w folderze wejściowym), komunikując się z kontenerami Docker i zapisując wynik bezpośrednio w docelowej strukturze plików, realizując w pełni autonomiczną automatyzację.
