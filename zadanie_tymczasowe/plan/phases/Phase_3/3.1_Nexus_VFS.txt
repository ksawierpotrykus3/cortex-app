# NEXUS V2 IMPLEMENTATION - PHASE 3.1
**Moduł:** Nexus Virtual File System (VFS) i Filtry ACL
**Cel 0-100%:** Zbudowanie muru pomiędzy surowym systemem operacyjnym a światem zewnętrznym, uniemożliwiając Agentom sięganie wprost do plików projektowych. Przez wirtualne ścieżki i białą listę autoryzacji system ma blokować ucieczki Directory Traversal.

---

## 🛠️ [ZADANIA DLA AI - CODERA]

1. **Wirtualne URI (`src/backend/security/vfs.ts`)**:
   - Stwórz abstrakcję `NexusVFS`. Nigdy nie podawaj surowej ścieżki `C:\Nexus_Workspace`.
   - Zmuszaj moduły wewnętrzne do korzystania ze wskaźników translacyjnych, np. wczytanie pliku odbywa się przez `nexus://workflow_logs/log_01`.
   - Wymuś stosowanie natywnego `path.resolve` i `path.normalize` przeciwko próbom ucieczek poza główny system plików za pomocą znaków wektorowych `../../`.
2. **Access Control List (ACL)**:
   - Skonstruuj instancję `AccessControl`. Podział opiera się na listach praw (`ALLOW` vs `DENY`).
   - Wprowadź politykę "Default Deny" (Domyślnie Zabronione). Agent logujący się do pliku X musi okazać explicitnie klucz tokenu wygenerowany dla jego DAG. Brak tokenu w VFS natychmiastowo rzuca uderzeniowy `ForbiddenVfsAccessException` niszcząc wywołanie RAG.

**[WERYFIKACJA DETERMINISTYCZNA (TDD DLA 2026)]**
- **Test 1:** Symuluj zapytanie RAG ze spreparowaną zmienną systemową `nexus://agents/../../../../../Windows/System32/config/SAM`. Test musi dowieść, że przed użyciem polecenia odczytu systemu, parser twardo ukrócił ścieżkę do wirtualnego koszyka, a wbudowany moduł normalizujący węzły nie pozwolił na ominięcie białej listy ACL wywalając asercję o złym formacie ataku.
